# Telegram Host App (external cPanel host — PHP 8.3)

اپ مستقل PHP برای دریافت مستقیم وب‌هوک تلگرام روی هاست خارج (سریع‌تر از رله Worker/پروکسی)، با کش محلی MySQL برای متن‌ها/کیبورد/کاتالوگ و فراخوانی زنده به سرور اصلی برای هر چیز حساس (OTP، ظرفیت، پرداخت).

## چرا این معماری سریع‌تر است

نسخه‌های قبلی («پروکسی» / Worker) هر آپدیت تلگرام را کامل به سرور اصلی forward می‌کردند — یعنی هر پیام یک hop شبکه‌ی اضافه (تلگرام → Worker/هاست → سرور اصلی → پاسخ) داشت. این اپ اکثر پاسخ‌ها (منو، لیست دوره/سمینار، متن‌ها) را **بدون هیچ تماسی با سرور اصلی**، مستقیماً از دیتابیس محلی همین هاست می‌دهد. فقط چیزهایی که واقعاً باید زنده باشند (OTP، ظرفیت سمینار، تخفیف) یک تماس API رمزنگاری‌شده به سرور اصلی می‌زنند.

## نصب روی cPanel

1. کل این پوشه (`telegram/`) را در هاست آپلود کنید (مثلاً در `~/telegram`).
2. یک دیتابیس MySQL بسازید و اسکیمای `db/schema.sql` را وارد کنید:
   ```bash
   mysql -u DB_USER -p DB_NAME < db/schema.sql
   ```
3. `config.sample.php` را کپی کنید به `config.php` و مقادیر را از پنل ادمین (Telegram → Infrastructure → حالت «هاست خارج») بردارید — پس از ذخیره تنظیمات آنجا، یک نسخه‌ی پُرشده آماده کپی نمایش داده می‌شود.
4. Document Root ساب‌دامین/فولدر را روی `telegram/public` بگذارید (نه روی ریشه `telegram/`) — یا اگر Document Root جداگانه ندارید، از `.htaccess` موجود در `public/` استفاده کنید و در پنل تلگرام آدرس را به همان مسیر بدهید.
5. یک Cron Job در cPanel اضافه کنید (هر ۵ دقیقه):
   ```
   */5 * * * * php /home/USER/telegram/cron/pull-sync.php >> /home/USER/telegram/cron.log 2>&1
   ```
   این دستور برای اولین‌بار هم باید دستی اجرا شود تا کش خالی نباشد:
   ```bash
   php cron/pull-sync.php
   ```
6. در پنل ادمین سایت، حالت اتصال را روی «هاست خارج» بگذارید و آدرس همین دامنه/ساب‌دامین را وارد کنید، سپس «ثبت وب‌هوک» را بزنید.

## معماری

```
Telegram → public/webhook.php → src/Bot.php
                                   ├─ src/Cache/SyncCache.php      (خوانش محلی — سریع)
                                   └─ src/Http/SyncClient.php      (تماس زنده رمزنگاری‌شده)
                                                │
                                                ▼
                              سرور اصلی: routes/telegram-host.php
                                    (HMAC + AES-256-GCM + Proxy-Origin)
```

## چه چیزی اینجا کامل پیاده‌سازی شده

- `/start` + ثبت‌نام با شماره موبایل (OTP زنده)
- منوی اصلی از کش محلی (بدون تماس با سرور در حالت عادی)
- لیست دوره‌ها/سمینارها از کش محلی
- بررسی ظرفیت سمینار — همیشه زنده، هرگز کش نمی‌شود

## چه چیزی هنوز باید تکمیل شود (خارج از دامنه‌ی این نسخه اول)

این نسخه‌ی اول عمداً کوچک نگه داشته شده تا امنیت (توکن پرداخت، رمز پیامک) هرگز روی هاست خارج قرار نگیرد. موارد زیر باقی مانده و باید در تکرارهای بعدی اضافه شوند:

- **پرداخت/سفارش نهایی**: فعلاً کاربر با دکمه inline به لینک سایت اصلی هدایت می‌شود؛ ایجاد سفارش و اتصال به درگاه زرین‌پال هنوز روی این هاست پیاده نشده (باید endpoint `order/create` جدید در سرور اصلی طراحی و اینجا مصرف شود).
- **پنل ادمین داخل ربات** (رویدادها، آمار، برودکاست) — فقط روی سرور اصلی است؛ اگر ادمین وب‌هوکش را به هاست بدهد، این بخش‌ها فعلاً پاسخ نمی‌دهند.
- **پشتیبانی/تیکت** و **کد تخفیف در مسیر خرید** — endpoint `discount/validate` در سرور آماده است ولی هنوز به جریان مکالمه‌ی این فایل وصل نشده.
- **required_chats enforcement** (عضویت اجباری کانال/گروه) — جدول کش موجود است؛ منطق بررسی عضویت (`getChatMember`) هنوز در `Bot.php` نوشته نشده.
- **Push فوری از سرور اصلی** به این هاست (مثلاً وقتی از پنل وب یک کاربر دستی ویرایش می‌شود) — فعلاً فقط pull دوره‌ای (کرون هر ۵ دقیقه) وجود دارد.

## امنیت

- توکن ربات، `webhook_secret`، `hmac_secret` و `aes_key` فقط در `config.php` (gitignored) — هرگز کامیت نشود.
- بدنه‌ی هر تماس sync با AES-256-GCM رمزنگاری و با HMAC-SHA256 امضا می‌شود (`src/Support/AesGcmCipher.php`, `src/Support/HmacClient.php`) — دقیقاً منطبق با معادل سمت سرور در `backend/app/Support/`.
- کلید کارت‌به‌کارت/زرین‌پال و اعتبارنامه‌ی SMS/OTP provider هرگز به این هاست فرستاده نمی‌شود.
